1.1 WORKFLOW EXECUTION ENGINE (15+ node types, 200+ docs concurrently)
--------------------------------------------------------------------------------

Q: Walk me through how the workflow engine works end-to-end.
A: "It's a DAG-based workflow executor. Workflows are defined as JSON
   configurations with nodes and edges — stored in MongoDB. When a document
   comes in, the engine parses the DAG, finds the start node, and executes
   nodes sequentially following the edges.

   Each node type has its own handler — there are conditional nodes for
   branching, loop nodes for iterating over line items, database lookup
   nodes, API call nodes, LLM nodes, and so on. The edges have 'success'
   and 'failure' handles, so the execution path is dynamic based on the
   data.

   Think of it as a configurable state machine — the same engine handles
   completely different document types just by swapping the workflow
   configuration."

Q: What are all 15+ node types? Explain a few.
A: "The main ones are:
   - Document Node (entry point)
   - Conditional Node (if/else branching with operators like equals,
     contains, regex, greater than, etc.)
   - ForEach Node (loops over arrays like line items, aggregates results)
   - Database Node (queries custom tables with variable templating)
   - Function Node (executes Python expressions with restricted builtins)
   - API Node (makes HTTP calls with dynamic URL/header/body)
   - Data Comparison Node (compares extracted data vs source data)
   - LLM Node (sends data to an LLM for validation or classification)
   - Approval Node (pauses the flow for human review)
   - Alert/Email/Slack Nodes (notifications)
   - And a few more for document transformation and status updates.

   For example, the Conditional Node evaluates a condition against the
   current flow context — if true, it follows the success edge to one
   subgraph, if false, it goes to another. This lets us handle different
   scenarios within the same workflow.

   The ForEach Node iterates over an array — like all line items in an
   invoice — runs child nodes for each item, and collects the aggregated
   results when done."

Q: How is concurrency handled for 200+ documents?
A: "We use an asyncio semaphore capped at 200 concurrent operations. Each
   document extraction acquires the semaphore before processing. Since
   the work is mostly I/O-bound — API calls to LLM providers, MongoDB
   reads/writes — async I/O lets us overlap these operations efficiently
   without needing multiple threads or processes."

Q: How do you handle errors in the workflow?
A: "There are two categories. Infrastructure errors — like an API timeout
   or database connection failure — these stop the flow execution and are
   logged for debugging. Then there are validation errors — like an amount
   mismatch or a missing field — these are collected and shown to the user
   but don't stop the flow.

   Everything is tracked in an execution trace, so if something goes wrong,
   we can see exactly which node failed and why."

Q: How are workflows defined? Can non-technical users create them?
A: "Workflows are declarative JSON — arrays of nodes and edges. But we also
   built a visual drag-and-drop editor on the frontend using React Flow.
   Users can drag node types onto a canvas, connect them with edges, and
   configure each node's parameters. The editor generates the JSON schema
   automatically and saves it. So yes, non-technical users can create and
   modify workflows without writing code."

Q: What's the difference between this and a simple if-else chain?
A: "This is a full DAG with handle-based routing — not linear. A conditional
   node can branch into completely different subgraphs. You can have loops,
   parallel validation paths, and dynamic execution based on the data. The
   execution path for one document can be completely different from another
   using the same workflow. It's more like a state machine than a script."

Q: How did you achieve ~90% cache hit rate?
A: "The 90% is specifically for config and template data — not all database
   queries. Template definitions and configuration data change very rarely,
   maybe once a day at most. So we cache them with a 24-hour TTL. Each
   request that would otherwise need 3-5 database round-trips for template
   and config lookups hits the cache instead. We measured this by comparing
   cache hits vs misses in our Redis metrics."

Q: How does real-time status update work for document processing?
A: "We use Server-Sent Events (SSE), not WebSockets. SSE is simpler for
   our use case — it's one-directional from server to client. The server
   maintains a queue per connection (max 100 messages). We send heartbeats
   to detect stale connections — if no heartbeat for 30 seconds or more
   than 50 messages are queued up, we mark the connection as unhealthy
   and clean it up. If a user disconnects, pending messages are buffered
   in Redis lists so they can be delivered when they reconnect."

Q: How does multi-tenancy work?
A: "Each tenant gets its own MongoDB database — the naming convention is
   tenant_{tenant_id}_{environment}. There's also a platform database
   for cross-tenant data like templates and user management. Tenant
   resolution happens at the auth layer — either from the Firebase JWT
   token or via organization_id query parameter. The codebase supports
   two versions: v1 uses tenant_id directly from the token, v2 maps
   organization_id to tenant_id."

Q: How does authentication work?
A: "Two auth mechanisms. Primary is Firebase JWT token verification —
   the frontend gets a token from Firebase, sends it in the Authorization
   header, and the backend verifies it using Firebase Admin SDK. We also
   support API key auth as an alternative path for system-to-system calls.
   Role-based access control is layered on top — users have roles, roles
   have permissions, and we check permissions on each endpoint."

Q: How do you handle cache invalidation?
A: "Pattern-based deletion using Redis SCAN — not KEYS, because SCAN is
   non-blocking. We scan in batches of 1000 keys, delete matching keys,
   and continue until done. We have specific invalidation methods for
   different data types — templates, validation flows, extraction modes,
   user sessions. After invalidation, we immediately cache the new version
   so the next request hits warm cache. There's also a nuclear option that
   invalidates all 16 cache patterns at once, but we rarely need it."

Q: What happens if Redis goes down?
A: "Redis operations are wrapped in try/except and fail silently — they
   return False or None without crashing the application. The system
   degrades gracefully to hitting MongoDB directly. It's slower, but
   functional. This was a deliberate design choice — caching should
   improve performance, not be a single point of failure."

Q: How does the MongoDB connection pool work?
A: "We use PyMongo with a connection pool — maxPoolSize of 50 and
   minPoolSize of 5. The driver manages connections automatically,
   checking them out for queries and returning them to the pool. This
   avoids the overhead of creating a new connection per request while
   keeping resource usage bounded."

Q: How do you serialize MongoDB ObjectIds for Kafka?
A: "MongoDB uses ObjectId for document IDs and datetime objects for
   timestamps. These aren't JSON-serializable by default. We use a
   custom JSON encoder that converts ObjectIds to strings and datetimes
   to ISO format strings before publishing to Kafka."

1.2 SAP ERP POSTING ENGINE (Non-PO, Rental, ZHPL)
--------------------------------------------------------------------------------

Q: What are the 3 SAP modes and how do they differ?
A: "Each mode handles a different type of document flow:

   Non-PO is for invoices without purchase orders. It has 3 sub-modes:
   NonPO1 does invoice-level matching with PO and POD documents. NonPO2
   handles scenarios where multiple rows map to the same invoice, with
   deduplication and PO utilization tracking. NonPO3 does full 3-way
   matching between Invoice, PO, and POD.

   Rental is for property lease billing — matching vendor agreements to
   invoices. It has a 3-tier vendor matching fallback: exact name match,
   cost center based matching, and fuzzy string matching with a 60%
   similarity threshold.

   ZHPL is for cost approval consolidation — it merges multiple cost
   center rows into single paired postings in SAP's specific format."

Q: How does debit/credit pair generation work?
A: "Every SAP posting needs balanced entries — a debit to an expense
   account and a credit to a vendor account under the same counter.
   The system takes the extracted invoice data, looks at the template
   configuration for which fields map to which SAP columns, and generates
   these pairs automatically.

   For example, in one mode, multiple invoice rows might map to the same
   posting counter — the debit is the sum of all GL amounts, and the
   credit is the total vendor amount. The key thing is that all the
   accounting rules are config-driven — different clients have different
   posting keys, GL accounts, and tax codes, and we handle all of that
   through configuration, not code changes."

Q: How does GST tax code resolution work?
A: "The system looks at the line items for IGST, CGST, and SGST amounts
   using field names configured per template. It calculates the effective
   tax rate — tax amount divided by base amount — and then looks up the
   corresponding SAP tax code from a configuration table. Nothing is
   hardcoded — if a client has different tax codes, we just update the
   config."

Q: How does vendor master lookup work in SAP posting?
A: "When generating SAP postings, we need the vendor's SAP account code
   and withholding tax code. The system looks up a vendor master table —
   first tries matching by vendor name using regex, falls back to matching
   by GSTIN. If multiple matches exist, we prefer the one with the main
   TDS code. The lookup result is cached per batch to avoid repeated
   database queries for the same vendor."

Q: How does duplicate invoice detection work?
A: "Before generating SAP postings, we check the processing history for
   the past 365 days for any invoice with the same number. If found, the
   posting gets flagged with a HOLD prefix in the remarks. This prevents
   double-payment — a critical requirement for enterprise accounting."

Q: How does the business place lookup work?
A: "SAP requires a business place code based on the state. We look up
   a business place table using regex matching on the state name. Each
   lookup is cached per row to avoid redundant database calls within
   the same batch."

Q: What about the ZHPL mode specifically?
A: "ZHPL is actually the simplest mode — it's pass-through. The user
   uploads a pre-formatted Excel file that already has the debit/credit
   pairs. Our code only renames the columns according to the SAP column
   mapping configuration. No database lookups needed — the data is
   already complete. We just sort by counter and posting key, apply the
   column mapping, and write the output Excel."

Q: Why not just hardcode the SAP format?
A: "Because enterprise clients have wildly different SAP configurations.
   Different posting keys, different GL account structures, different tax
   codes. If we hardcoded it, onboarding a new client would mean changing
   code. With our template-based approach, new clients are onboarded by
   changing configuration only. That's critical for a multi-tenant SaaS
   product — you can't deploy different code per client."

1.3 APACHE KAFKA PIPELINES (6+ microservices)
--------------------------------------------------------------------------------

Q: How is Kafka used in your system?
A: "We have 6 topics forming an event-driven pipeline. When a document is
   uploaded, a message goes to the 'document-uploaded' topic. The extraction
   service consumes this, processes the document with an LLM, and publishes
   to 'extraction-complete'. Similarly, there are topics for template
   creation requests and completions, and validation flow creation.

   Each microservice has its own consumer group, so each service gets its
   own copy of every message. Within a service, Kafka distributes partitions
   across instances for horizontal scaling."

Q: How do you achieve exactly-once semantics?
A: "Honestly, we do at-least-once with idempotent processing, not true
   exactly-once. On the producer side, we set acks=all so all replicas
   acknowledge. On the consumer side, we do frequent offset commits. And
   at the application level, we track processed document IDs to avoid
   reprocessing if a consumer restarts.

   True exactly-once in Kafka requires the transactional API with
   idempotent producers — we didn't need that level of guarantee since
   our processing is naturally idempotent."

Q: Tell me about the batch publishing with compression.
A: "We configure the producer with gzip compression and a 100ms linger
   time. That means the producer waits up to 100ms to batch multiple
   messages together before sending. Combined with compression, this
   significantly reduces network overhead — instead of sending many small
   messages individually, we send fewer compressed batches."

Q: What happens when a consumer fails?
A: "The consumer sends heartbeats every second. If Kafka doesn't receive
   a heartbeat for 30 seconds, it triggers a consumer group rebalance
   and reassigns that consumer's partitions to other healthy consumers.
   Failed messages are logged and the consumer moves on. We don't have
   a formal dead letter queue right now — errors are tracked in
   application state."

Q: How would you add a Dead Letter Queue?
A: "I'd create a separate 'dead-letter' topic. In the consumer, after
   N retries of a failed message, publish it to the DLQ topic with error
   metadata — timestamp, exception, retry count. Then a separate monitoring
   service or dashboard consumes the DLQ for alerting and manual
   reprocessing. It's a pattern we'd add as the system grows."

Q: How does Kafka authentication work in your system?
A: "We use SASL_PLAINTEXT authentication. The producer and consumer
   authenticate with the Kafka broker using username/password credentials.
   This is standard for internal services where the network is already
   trusted but you still want access control."

Q: What's the difference between send() and send_and_wait() in your producer?
A: "We use both depending on criticality. For document upload events,
   we use fire-and-forget send() — it returns a future but we don't
   await it, so the API response isn't blocked. For critical events
   like extraction completion, we use send_and_wait() which blocks
   until all replicas acknowledge. This gives us reliability guarantees
   for events that drive downstream processing."

Q: How do you handle batch publishing?
A: "For bulk operations like publishing multiple documents, we fire all
   the send futures at once, then await each one. This gives us
   parallelism in the publishing step while still confirming delivery.
   The producer's linger_ms=100 helps batch these into fewer network
   calls automatically."

Q: Kafka vs RabbitMQ — when to use which?
A: "We actually use both in different projects. Kafka for the document
   pipeline because we need strong ordering, replay capability, and
   multiple consumers reading the same events. RabbitMQ for the job
   application task queue because it has better support for flexible
   routing, priority queues, and per-task acknowledgment — classic
   task queue patterns."

1.4 MULTI-MODEL LLM EXTRACTION (GPT-4o, Gemini, Claude)
--------------------------------------------------------------------------------

Q: How does multi-model routing work?
A: "Different LLM providers have different strengths and input requirements.
   The model is configured per template — so one document type might use
   GPT-4o while another uses Gemini. The routing logic handles the
   differences: some models need PDF-to-image conversion first (OpenAI
   vision models), while others accept PDFs directly (Gemini, Claude).

   This gives us flexibility — if one model works better for a particular
   document format, we just change the template config."

Q: How does prompt templating work?
A: "We have a base extraction prompt that defines the output format —
   structured key-value data plus table data. A prompt builder dynamically
   constructs the field list by merging field names with their descriptions
   from the template configuration. So the model knows exactly what to
   extract and what each field means. This is all driven by the template —
   adding a new document type means defining new fields and descriptions,
   not changing the prompt logic."

Q: How do you handle rate limiting across models?
A: "We have a rate limiter that manages API quotas per provider using Redis
   for distributed state. Even with multiple extraction service instances
   running, the total API calls stay within provider limits. This works
   alongside the asyncio semaphore for concurrency control — the semaphore
   limits how many documents we process at once, and the rate limiter
   ensures we don't exceed API quotas."

Q: How is 95%+ accuracy measured?
A: "Field-level accuracy — we compare extracted values against manually
   verified ground truth documents. The validation workflow checks each
   extracted field against expected values. Fields that pass validation
   are considered accurate. The 95%+ figure is across our production
   document corpus for templates that have been properly configured."

Q: What happens when extraction fails?
A: "Failed extractions are tracked with a FAILED status. The system
   excludes failed documents from downstream matching to avoid propagating
   errors. For low-confidence results, the validation workflow can flag
   them for manual review through approval nodes — a human reviews and
   corrects before the data moves forward."

Q: How do you handle different document formats?
A: "The template system abstracts document formats. Each template defines
   what fields to extract (structured header data + table line items),
   field descriptions for the LLM prompt, and which model works best for
   that document type. Supporting a new format means creating a new
   template — no code changes needed. Templates can be created manually
   via the visual editor or automatically via the LangGraph agent system."

Q: How does the extraction service communicate with the main backend?
A: "Through Kafka. The main backend publishes extraction requests to a
   topic. The extraction microservice consumes these, processes documents
   with the configured LLM, and publishes completion events. The main
   backend's consumer picks these up and stores results in MongoDB. This
   decoupling means the extraction service can scale independently and
   we can update models without touching the main backend."

Q: What's the default LLM model?
A: "Claude Sonnet is our current default for template creation, but the
   model is configurable per template creation request. For extraction,
   each template can specify its preferred model — some documents extract
   better with GPT-4o vision, others work fine with Gemini. This
   per-template model selection was informed by testing different models
   on different document types."

1.5 LANGGRAPH-BASED AI AGENTS (template auto-generation)
--------------------------------------------------------------------------------

Q: How does the multi-agent system work?
A: "There are 3 specialized agents in a LangGraph workflow:

   First, a PDF Analysis Agent analyzes the document structure and
   identifies what fields exist and how they relate to each other.

   Second, a Template Creation Agent takes that analysis and generates
   the extraction template — the field definitions, data types, and
   descriptions.

   Third, a Validation Flow Agent designs the validation workflow —
   what checks to run on the extracted data.

   These agents are orchestrated as a 9-node DAG in LangGraph. Each
   agent's output feeds into the next through shared state. The whole
   pipeline takes a PDF and produces a ready-to-use template with
   validation rules."

Q: How do agents collaborate?
A: "Through shared state. LangGraph maintains a state object that each
   agent reads from and writes to. The PDF Analysis Agent adds the
   document structure, the Template Creation Agent reads that and adds
   the template schema, and so on. There's also a conversation memory
   manager for the interactive chat interface."

Q: There's a chat interface too?
A: "Yes, we also built conversational agents for guided template creation.
   There's an intent router that figures out what the user wants, then
   routes to specialized agents — document analyzer, template creator,
   validation flow creator, or a general assistant. Non-technical users
   can create templates by describing what they need in natural language."

Q: What is LangGraph and how is it different from LangChain?
A: "LangChain is a framework for building LLM applications — it gives you
   chains for sequential operations and agents for tool-using. LangGraph
   extends that by adding graph-based execution. Nodes are functions or
   agents, edges define control flow, and importantly, you can have cycles
   — loops. This enables multi-step agent workflows with proper state
   management, conditional branching, and human-in-the-loop patterns.
   It's essentially a state machine for AI agents."

1.6 DOCUMENT VALIDATION + 3-WAY MATCHING (Invoice vs PO vs POD)
--------------------------------------------------------------------------------

Q: Walk me through the 3-way matching algorithm.
A: "The goal is to match each invoice against its Purchase Order and
   Proof of Delivery to validate correctness.

   Step 1: Build lookup dictionaries from all three document types,
   keyed by their respective identifying numbers — invoice number,
   PO number, etc.

   Step 2: For each invoice, find its matching PO by PO number. Track
   how much of the PO value has been utilized across all invoices.

   Step 3: Find the matching POD. This is where it gets interesting —
   we try multiple matching strategies:
   - First, direct match by invoice number
   - If that fails, extract the trailing numeric digits and try a
     partial match (because different systems format invoice numbers
     differently — 'INV-2024-12345' vs just '12345')
   - If that also fails, fall back to matching by PO number

   Step 4: Once matched, resolve derived fields like supply state
   using a priority chain across the three documents."

Q: Why the numeric suffix matching? Sounds like a hack.
A: "It's a real-world necessity. In enterprise systems, the same invoice
   number gets formatted differently across systems. The vendor's invoice
   says 'INV-2024-12345', the POD system records it as '12345', and the
   PO system has something else entirely. We discovered this when a major
   client's matching was failing — the numbers were the same, just with
   different prefixes. Extracting the trailing digits solved it cleanly."

Q: What fuzzy matching do you use?
A: "Python's SequenceMatcher from difflib. We compute similarity ratios
   between strings — typically with a 70% threshold. Main use case is
   vendor name matching in rental mode, where names have slight variations
   like 'ABC Pvt Ltd' vs 'ABC Private Limited'. We chose SequenceMatcher
   over libraries like thefuzz because it's in the standard library —
   no extra dependency — and good enough for our use case."

1.7 APPLYSURGE - AUTONOMOUS JOB APPLICATION PLATFORM
--------------------------------------------------------------------------------

Q: How does the browser automation work?
A: "We use Playwright with a browser pool pattern — a singleton manager
   that controls up to 5 concurrent browser contexts using an asyncio
   semaphore. Each context gets a fresh identity — random user agent from
   7 modern Chrome/Firefox versions, random viewport from 5 sizes, and
   geolocation set to the user's city.

   The key is the 8 stealth scripts injected into every page:
   1. Override navigator.webdriver to undefined
   2. Fake the Chrome object
   3. Override permissions query
   4. Randomize canvas fingerprint (shift RGB values by -5 to +5)
   5. Spoof WebGL vendor/renderer to Intel
   6. Fake plugins and language arrays
   7. Override hardware concurrency
   8. Override platform strings

   We also launch real Chrome (not bundled Chromium) with automation
   flags disabled. The combination makes it nearly impossible for bot
   detection systems to distinguish from a real user."

Q: How do you simulate human-like behavior?
A: "Several techniques. Typing has random delays between 50-150ms per
   character, with a 5% chance of a 'thinking pause' of 300-800ms mid-
   typing. Page scrolling happens in 3-6 randomized steps with pauses
   between each scroll increment. Clicks have random delays of 0.1-0.5
   seconds before clicking and 0.5-1.5 seconds after. All these small
   randomizations make the interaction pattern look natural."

Q: How does the distributed task pipeline work?
A: "Celery with RabbitMQ as the broker and Redis as the result backend.

   timeout. So tasks ack immediately on receipt. If a worker crashes,
   we detect it via the session lock mechanism.

   Workers are configured with prefetch_multiplier=1 (one task at a time
   for browser work), max_tasks_per_child=100 (restart after 100 tasks
   to prevent memory leaks), and max_memory_per_child=512MB.

   Celery Beat handles scheduled jobs — daily limit resets at midnight,
   summary emails at 8pm, stats update at 11:55pm, startup discovery

Q: How does AI-powered job matching work?
A: "LLM-based scoring with 5 weighted components:

Q: How do you ensure only one automation session runs per user?
A: "Atomic Redis SET with NX (set-if-not-exists) and EX (expiry) flags.
   The key is per-user, the TTL is 3900 seconds (65 minutes — matching
   the task hard time limit of 1 hour plus 5 minute buffer).

   Only one caller can acquire the lock because SET NX is atomic. If
   the lock exists, the new request is rejected immediately. When the
   session completes normally, we explicitly delete the lock. If it
   crashes, the TTL ensures it auto-expires.

   There's also a stop signal mechanism — the user can click 'Stop Agent',
   which sets a key in Redis. The running task periodically checks for
   this key using GETDEL (atomic get-and-delete) so only the running
   worker sees the signal."

Q: How is real-time tracking implemented?

Q: How does session persistence work across automation runs?
A: "Browser cookies are encrypted with Fernet (AES-128-CBC + HMAC)

   look suspicious and trigger security alerts.

   The encryption key is derived from a config secret using SHA-256.
   All sensitive data — platform passwords, API keys, OAuth tokens —

Q: How does the cold email pipeline work?

Q6: Design a Rate Limiter
--------------------------------------------------------------------------------

Q8: CAP Theorem — how does it apply to your stack?
--------------------------------------------------------------------------------
A: Consistency, Availability, Partition tolerance — you can only guarantee
   two out of three in a distributed system.

   - MongoDB (default): CP — consistent reads from primary, may lose
     availability during primary election
   - Redis: AP — available and partition-tolerant, eventual consistency
     in cluster mode
   - PostgreSQL: CA — strong consistency and availability, but doesn't
     handle network partitions natively (single node)
   - Kafka: CP — consistent writes with acks=all, may be unavailable
     if not enough replicas are in sync

   "In practice, it's a spectrum, not a binary choice. You configure the
   trade-off — like MongoDB read preference or Kafka acks level."

--------------------------------------------------------------------------------
3.1 OPERATING SYSTEMS
--------------------------------------------------------------------------------

Q: Process vs Thread?
A: Process has its own memory space — heap, stack, everything isolated.
   Creating a process is heavy, but crashes don't affect other processes.
   Communication between processes needs IPC mechanisms.

   Thread shares memory within its parent process — same heap, but each
   thread has its own stack. Lighter and faster to create. But a crash
   in one thread can take down the entire process.

   When to use which: Threads for I/O-bound tasks within an application.
   Processes for CPU-bound parallelism or when you need isolation.

Q: What is a deadlock? 4 necessary conditions?
A: Deadlock is when two or more processes are stuck waiting for each other
   forever. All 4 conditions must hold simultaneously:
   1. Mutual Exclusion — resource can only be held by one process
   2. Hold and Wait — process holds one resource while waiting for another
   3. No Preemption — can't forcibly take a resource from a holder
   4. Circular Wait — A waits for B, B waits for A

   Prevention: break any one condition. Most practical: impose a resource
   ordering to break circular wait. Banker's algorithm for avoidance.

Q: What is virtual memory?
A: An abstraction that gives each process the illusion of having its own
   large, contiguous address space. Uses disk as extension of RAM.
   Implemented via paging — memory divided into fixed-size pages (4KB
   typically). A page table maps virtual addresses to physical addresses.
   The TLB (Translation Lookaside Buffer) caches recent translations
   for performance.

Q: Explain page faults.
A: When a process accesses a page that's not currently in RAM:
   1. CPU raises a page fault interrupt
   2. OS finds the page on disk (swap space)
   3. Loads it into a free physical frame (may evict another page using
      LRU, FIFO, or Clock algorithm)
   4. Updates the page table mapping
   5. Resumes the instruction

   Minor page fault: page is in memory but not mapped. Major page fault:
   page must be loaded from disk — much slower.

Q: What is a context switch? Why is it expensive?
A: Saving the entire state (registers, program counter, stack pointer,
   page table) of the current process and loading another's.

   Expensive because:
   - Register save/restore
   - TLB flush — cached address translations become invalid
   - CPU pipeline flush
   - Cache cold start — L1/L2 cache misses after the switch

   Thread switches within the same process are cheaper because they
   share the same address space — no TLB flush needed.

Q: Mutex vs Semaphore?
A: Mutex is a binary lock with ownership — only the thread that locked
   it can unlock it. Used for mutual exclusion.

   Semaphore is a counter — can allow N concurrent accesses (counting
   semaphore). No ownership — any thread can signal. Binary semaphore
   (count=1) is similar to mutex but without ownership.

   In my projects, I use asyncio.Semaphore for concurrency limiting —
   capping concurrent API calls to 200.

Q: CPU Scheduling Algorithms?
A: - FCFS: Simple but has convoy effect — short jobs stuck behind long ones
   - SJF: Optimal average wait time, but impractical — you'd need to
     predict burst times. Long jobs can starve.
   - Round Robin: Time quantum, fair for interactive systems. Too small
     quantum = context switch overhead, too large = becomes FCFS.
   - Priority Scheduling: Higher priority runs first. Starvation solved
     by aging — gradually increase priority of waiting processes.
   - MLFQ: Multiple queues with different priorities and time slices.
     Real-world schedulers (Linux CFS) use variants of this.

Q: What is thrashing?
A: When the system spends more time swapping pages in and out of memory
   than actually executing processes. Happens when too many processes
   compete for limited RAM — each causes page faults, evicting pages
   other processes need. Solution: reduce the number of active processes,
   or use the working set model to ensure each process has enough frames.

Q: User mode vs Kernel mode?
A: User mode has restricted access — can't directly access hardware or
   other processes' memory. Kernel mode has full access to everything.

   System calls (fork, read, write, open) transition from user mode
   to kernel mode via a trap/interrupt. A mode bit in the CPU status
   register controls which mode is active. This separation is what
   prevents user programs from crashing the OS.

Q: Internal vs External fragmentation?
A: Internal: wasted space within an allocated block. Process needs 30KB,
   gets a 32KB partition — 2KB wasted. Happens with fixed-size allocation.

   External: total free memory is sufficient but scattered in small
   non-contiguous chunks. No single block is big enough. Happens with
   variable-size allocation.

   Solutions: Compaction (move processes to consolidate free space),
   Paging (eliminates external fragmentation entirely by using fixed
   pages).

Q: What are zombie and orphan processes?
A: Zombie: a process that has finished execution but its parent hasn't
   read its exit status yet (via wait()). Takes up a process table entry
   but no resources. Too many zombies can exhaust the process table.

   Orphan: a process whose parent has terminated. The init/systemd process
   adopts orphans and cleans them up. Not as problematic as zombies.

Q: IPC mechanisms?
A: - Pipes: unidirectional byte stream between parent-child processes
   - Named Pipes (FIFOs): like pipes but with a filesystem name,
     usable between unrelated processes
   - Message Queues: structured messages with types and priorities
   - Shared Memory: fastest — processes map the same memory region.
     Needs synchronization (semaphores/mutexes)
   - Sockets: for network IPC, works across machines
   - Signals: async notifications (SIGKILL, SIGTERM, etc.)

--------------------------------------------------------------------------------
3.2 DBMS
--------------------------------------------------------------------------------

Q: What are ACID properties?
A: Atomicity: Transaction is all-or-nothing. If any step fails, everything
   rolls back. No partial updates.

   Consistency: Transaction brings the database from one valid state to
   another. All constraints (foreign keys, unique, check) satisfied.

   Isolation: Concurrent transactions don't interfere with each other.
   Each sees a consistent snapshot as if it were running alone.

   Durability: Once committed, the data survives even if the system
   crashes. Typically implemented via Write-Ahead Logging (WAL).

Q: Explain normalization (1NF through BCNF).
A: 1NF: All values are atomic — no arrays or repeating groups in columns.

   2NF: 1NF + no partial dependencies. Every non-key column depends on
   the ENTIRE primary key, not just part of it (relevant for composite
   keys).

   3NF: 2NF + no transitive dependencies. Non-key columns don't depend
   on other non-key columns.

   BCNF: Stricter than 3NF. Every determinant is a candidate key. Handles
   edge cases 3NF misses.

   Goal: reduce redundancy and prevent update/insert/delete anomalies.

Q: When would you denormalize?
A: Read-heavy workloads like analytics dashboards, reducing complex JOINs,
   when write frequency is low compared to reads. Trade-off: faster reads
   but slower writes, potential inconsistency, more storage.

   In my projects, MongoDB collections are inherently denormalized —
   we embed extracted data as subdocuments to avoid JOINs entirely.
   Makes sense when data is always accessed together.

Q: B-Tree vs Hash Index?
A: B-Tree: Balanced tree. Supports both equality AND range queries.
   O(log n) lookup. Default in PostgreSQL and MongoDB. Data is sorted,
   so good for ORDER BY, BETWEEN, and range scans.

   Hash: O(1) for exact equality lookups. No range support at all.
   Faster for point queries but very limited. Used in hash partitioning,
   in-memory structures like Redis.

Q: Transaction Isolation Levels?
A: From weakest to strongest:

   Read Uncommitted: Can see uncommitted changes (dirty reads). Fastest.
   Read Committed: Only sees committed data. Default in PostgreSQL.
   Repeatable Read: Same query returns same results within a transaction.
     Default in MySQL InnoDB.
   Serializable: Full isolation, as if transactions ran one-by-one.
     Slowest, uses locks or snapshot isolation.

   Each level prevents more anomalies:
   - Read Committed prevents dirty reads
   - Repeatable Read additionally prevents non-repeatable reads
   - Serializable additionally prevents phantom reads

Q: SQL vs NoSQL — when to use which?
A: SQL (PostgreSQL): When you need structured data with relationships,
   complex queries with JOINs, ACID transactions, strong consistency.
   I used PostgreSQL for EventZity — 26+ tables with relationships.

   NoSQL (MongoDB): When schema varies across records, you need
   horizontal scaling, high write throughput, or nested documents.
   I used MongoDB for document extraction because each template extracts
   different fields — flexible schema was essential.

   Redis (Key-Value): Caching, sessions, real-time features, leaderboards.

   "The choice depends on the data model and access patterns, not
   personal preference."

Q: What is sharding?
A: Horizontal partitioning of data across multiple database instances.

   Strategies:
   - Range-based: Split by value range (users 1-1M on shard 1). Simple
     but can create hotspots if distribution is uneven.
   - Hash-based: Hash the key to determine the shard. Even distribution
     but can't do range queries across shards efficiently.
   - Directory-based: Lookup table maps keys to shards. Flexible but the
     lookup table becomes a bottleneck and single point of failure.

   Challenges: Cross-shard queries are expensive, rebalancing is complex,
   and maintaining consistency across shards is hard.

Q: Clustered vs Non-clustered Index?
A: Clustered: Data rows are physically sorted by the index key on disk.
   Only ONE per table. Range queries are very fast because data is
   contiguous. In PostgreSQL, the primary key effectively acts as this.

   Non-clustered: Separate data structure with pointers to the actual
   rows. You can have multiple. Lookup requires an extra hop — go to
   index, find pointer, follow pointer to data.

Q: Optimistic vs Pessimistic Locking?
A: Pessimistic: Lock the resource when you start working on it. Others
   wait. Guarantees no conflicts but reduces concurrency. Use when
   conflicts are likely.

   Optimistic: Don't lock — just check at commit time if anyone else
   modified the data (via version number or timestamp). If conflict,
   retry. Better concurrency but more work on conflicts. Use when
   conflicts are rare.

Q: What is WAL (Write-Ahead Logging)?
A: Before modifying actual data files, write the change to a log first.
   If the system crashes mid-write, replay the log to recover. This is
   how databases guarantee durability. PostgreSQL uses WAL for both
   crash recovery and streaming replication (send WAL segments to
   replicas).

Q: Explain EXPLAIN ANALYZE.
A: Shows the query execution plan with actual runtime statistics.
   Key things to look for:
   - Sequential Scan on a large table = probably needs an index
   - Big gap between estimated vs actual rows = stale statistics
   - Nested Loop with Sequential Scan = potential N+1 problem
   - Sort on large dataset without index = expensive

   Always run ANALYZE to update statistics after bulk operations.

--------------------------------------------------------------------------------
3.3 NETWORKING
--------------------------------------------------------------------------------

Q: What happens when you type a URL in the browser? (THE classic question)
A: 1. Browser checks caches: browser cache, OS cache, router, ISP
   2. DNS resolution: recursive query through root -> TLD -> authoritative
      nameservers to get the IP address
   3. TCP connection: 3-way handshake (SYN, SYN-ACK, ACK)
   4. TLS handshake if HTTPS: certificate exchange, key agreement
   5. HTTP request: GET /path with headers
   6. Server processes the request, returns HTTP response
   7. Browser parses HTML, discovers CSS/JS/image resources
   8. Fetches additional resources (may use HTTP/2 multiplexing)
   9. Renders: DOM tree -> CSSOM -> Render tree -> Layout -> Paint

Q: TCP vs UDP?
A: TCP: Connection-oriented, reliable, ordered delivery, flow control,
   congestion control. 3-way handshake setup. Used for HTTP, SSH,
   database connections — anything where you can't afford data loss.

   UDP: Connectionless, no guarantees, no ordering, no handshake.
   Faster, lower overhead. Used for DNS, video streaming, gaming,
   VoIP — where speed matters more than perfect delivery.

Q: Explain the TCP 3-way handshake.
A: 1. Client sends SYN with a sequence number
   2. Server responds with SYN-ACK, acknowledging client's sequence
      and sending its own
   3. Client sends ACK, acknowledging server's sequence

   Connection is now established. Both sides agree on sequence numbers
   for ordering.

   Teardown is 4-way: FIN -> ACK -> FIN -> ACK (each side closes
   independently).

Q: HTTP/1.1 vs HTTP/2 vs HTTP/3?
A: HTTP/1.1: One request per TCP connection (or pipelining, rarely used).
   Head-of-line blocking — slow response blocks subsequent ones.
   Workaround: browsers open 6 parallel connections.

   HTTP/2: Multiplexing — multiple requests over single TCP connection.
   Header compression (HPACK). Server push. Binary protocol instead
   of text. Still suffers from TCP-level head-of-line blocking.

   HTTP/3: Uses QUIC (over UDP instead of TCP). Eliminates TCP-level
   head-of-line blocking. Built-in encryption. Faster connection
   setup (0-RTT). Still rolling out widely.

Q: REST vs GraphQL?
A: REST: Resource-based URLs, fixed response shapes, multiple endpoints.
   Can over-fetch (get all fields when you need 3) or under-fetch
   (need multiple requests). Easy to cache via HTTP caching.

   GraphQL: Single endpoint, client specifies exact fields needed.
   Reduces round trips, no over-fetching. Harder to cache, more complex
   server. Better for mobile apps with bandwidth constraints.

   All my APIs are REST — simpler, well-understood, good tooling.

Q: What are WebSockets?
A: Full-duplex bidirectional communication over a single TCP connection.
   Starts as an HTTP upgrade request (101 Switching Protocols). Once
   established, either side can send messages without waiting.

   Used for real-time features: chat, live updates, gaming. I use
   Socket.IO in my projects, which wraps WebSocket with fallback to
   HTTP long-polling if WebSocket isn't available.

Q: How does DNS work?
A: Hierarchical resolution:
   1. Browser/OS cache check
   2. Recursive resolver (usually ISP's) queries root nameserver
   3. Root returns TLD nameserver (.com, .org, .in)
   4. TLD returns authoritative nameserver for the domain
   5. Authoritative nameserver returns the actual IP address

   Caching at every level with TTL. This is why DNS changes take time
   to propagate — caches hold stale entries until TTL expires.

Q: What is a CDN?
A: Content Delivery Network. Caches static content (images, CSS, JS) at
   edge servers geographically close to users. Reduces latency, offloads
   the origin server, improves availability. CloudFront, Cloudflare are
   examples. Dynamic content usually still goes to the origin.

Q: Load Balancing algorithms?
A: - Round Robin: distribute equally in rotation
   - Weighted Round Robin: stronger servers get more traffic
   - Least Connections: route to server with fewest active connections
   - IP Hash: same client always goes to same server (sticky sessions)
   - Consistent Hashing: minimizes redistribution when servers
     are added/removed

   L4 (transport layer): fast, routes based on IP/port.
   L7 (application layer): smarter, can route based on URL path,
   headers, cookies.

Q: What is CORS?
A: Cross-Origin Resource Sharing. Browser security mechanism that
   prevents scripts on one domain from making requests to another domain.
   Server must explicitly allow it via Access-Control-Allow-Origin headers.

   For non-simple requests (PUT, DELETE, custom headers), the browser
   sends a preflight OPTIONS request first. I configure CORS in FastAPI
   and Express middleware.

Q: Cookies vs Sessions vs JWT?
A: Cookies: Small data stored in browser, sent with every request. Can
   be HttpOnly (no JS access), Secure (HTTPS only), SameSite.

   Sessions: Server-side state storage. Cookie holds just the session ID,
   actual data is on the server. More secure but requires server memory
   or a session store (Redis).

   JWT: Self-contained token with encoded claims. Stateless — server
   doesn't need to store anything. But can't be easily revoked (no
   server-side invalidation without a blocklist). Good for distributed
   systems where sharing session state across servers is hard.

Q: Forward proxy vs Reverse proxy?
A: Forward proxy: Sits in front of CLIENTS. Client sends requests through
   the proxy. Used for anonymity, filtering, caching. Example: corporate
   proxy servers.

   Reverse proxy: Sits in front of SERVERS. Receives client requests and
   forwards to appropriate backend server. Used for load balancing, SSL
   termination, caching. Example: Nginx, HAProxy.

Q: What are HTTP status codes? (common ones)
A: 200 OK — success
   201 Created — resource created (POST)
   301 Moved Permanently — permanent redirect (cacheable)
   302 Found — temporary redirect (not cached)
   400 Bad Request — client error, invalid input
   401 Unauthorized — authentication required
   403 Forbidden — authenticated but not authorized
   404 Not Found — resource doesn't exist
   429 Too Many Requests — rate limited
   500 Internal Server Error — server-side failure
   502 Bad Gateway — upstream server returned invalid response
   503 Service Unavailable — server overloaded or in maintenance

--------------------------------------------------------------------------------
3.4 OOP
--------------------------------------------------------------------------------

Q: 4 pillars of OOP?
A: Encapsulation: Bundle data and methods together, hide internal state
   via access modifiers. Only expose what's necessary.

   Abstraction: Expose a clean interface, hide implementation details.
   Users of a class don't need to know how it works internally.

   Inheritance: Code reuse via parent-child relationships. "is-a"
   relationship. Dog is an Animal.

   Polymorphism: Same interface, different behavior. Method overriding
   (runtime — subclass provides its own implementation) and overloading
   (compile-time — same method name, different parameters).

Q: SOLID principles?
A: S — Single Responsibility: A class should have only one reason to
   change. One job per class.

   O — Open/Closed: Open for extension, closed for modification. Add
   new behavior without changing existing code.

   L — Liskov Substitution: Subclasses should be usable wherever the
   parent class is expected, without breaking behavior.

   I — Interface Segregation: Many small, focused interfaces are better
   than one large, general-purpose interface.

   D — Dependency Inversion: Depend on abstractions (interfaces), not
   concrete implementations. High-level modules shouldn't depend on
   low-level modules.

Q: Common Design Patterns?
A: Singleton: One instance globally. Use for DB connection pools, loggers,
   config objects. Drawback: hidden global state, hard to test.

   Factory: Creates objects without exposing creation logic. Client asks
   for an interface, factory decides the concrete class. Decouples
   creation from usage.

   Observer: Publisher maintains list of subscribers, notifies them on
   state change. Kafka is essentially pub/sub — producers publish,
   consumers observe.

   Strategy: Define a family of algorithms, encapsulate each one, make
   them interchangeable. The algorithm varies independently from the
   client. Example: different rate limiting strategies.

   Builder: Construct complex objects step-by-step. Useful when a
   constructor would have too many parameters.

Q: Inheritance vs Composition?
A: Inheritance: "is-a" relationship. Tight coupling — changing the parent
   can break children. Fragile base class problem.

   Composition: "has-a" relationship. More flexible — swap components at
   runtime. Loose coupling.

   "Favor composition over inheritance." In my workflow engine, each node
   type is a separate class composed into the executor, not inheriting
   from a deep hierarchy.

Q: What is Dependency Injection?
A: Instead of a class creating its own dependencies, they're provided
   from the outside — usually via the constructor.

   Benefits: easier testing (inject mocks), loose coupling (swap
   implementations), clearer dependencies (explicit in constructor).

   In FastAPI, the Depends() system injects database sessions, auth
   objects, and configs into route handlers automatically.

--------------------------------------------------------------------------------
4.1 APACHE KAFKA
--------------------------------------------------------------------------------

Q: What is Kafka?
A: Distributed event streaming platform. Core concepts:
   - Producers publish messages to topics
   - Consumers read messages from topics
   - Brokers are the Kafka servers that store data
   - Topics are logical channels, split into partitions
   - Partitions enable parallelism and ordering within a partition
   - Consumer Groups coordinate consumers for parallel processing

Q: How do partitions work?
A: Topics are split into partitions for parallelism. Each partition is
   an ordered, immutable, append-only log. Messages have sequential
   offsets. A partition key (hash-based) determines which partition a
   message goes to. IMPORTANT: Ordering is guaranteed only within a
   single partition, not across partitions. More partitions = more
   parallelism, up to one consumer per partition.

Q: What are offsets?
A: A sequential ID assigned to each message within a partition. Consumers
   track their position via committed offsets (stored in a special
   __consumer_offsets topic). Auto-commit commits periodically, manual
   commit gives you control. If a consumer crashes after processing but
   before committing — the message gets reprocessed.

Q: Exactly-once semantics?
A: Three delivery levels:
   - At-most-once: Send and forget. May lose messages.
   - At-least-once: Ack after receive. May duplicate. Most common.
   - Exactly-once: Idempotent producer (assigns sequence numbers to
     detect duplicates) + transactional API. Consumer reads only
     committed messages. Needs Kafka Streams or manual transaction
     management.

Q: What happens during rebalancing?
A: Triggered when a consumer joins, leaves, or is considered dead.
   The group coordinator reassigns partitions. During rebalance,
   consumption PAUSES for the entire group.
   Two strategies: Eager (stop all consumers, reassign everything) vs
   Cooperative/Incremental (only reassign affected partitions — much
   less disruptive).

Q: What is log compaction?
A: A retention policy that keeps only the latest value per key (instead
   of keeping all messages for a time period). Useful for changelog
   topics where you only care about current state. A background thread
   compacts old segments, removing superseded entries.

Q: How to handle consumer lag?
A: Monitor via consumer group lag metrics. Solutions:
   1. Add more consumers (up to the number of partitions)
   2. Increase partition count (careful — can't decrease)
   3. Optimize processing logic (batch processing, async I/O)
   4. Increase consumer fetch batch sizes
   5. Find and fix the bottleneck in the slow consumer

--------------------------------------------------------------------------------
4.2 REDIS
--------------------------------------------------------------------------------

Q: Why is Redis so fast?
A: - In-memory storage (no disk I/O for reads)
   - Single-threaded command execution (no lock contention)
   - Optimized C data structures (skip lists, hash tables)
   - I/O multiplexing (epoll on Linux, kqueue on macOS)
   - Simple wire protocol (RESP)
   Redis 6+ uses I/O threads for network operations, but command
   execution remains single-threaded.

Q: Data structures in Redis?
A: Strings, Lists (linked list), Sets (unique unordered), Sorted Sets
   (scored members — great for leaderboards), Hashes (field-value maps),
   Bitmaps, HyperLogLog (cardinality estimation), Streams (append-only
   log — like a mini Kafka), Geospatial indexes.

Q: RDB vs AOF persistence?
A: RDB: Point-in-time snapshots. Compact file, fast to restore. But
   potential data loss between snapshots.

   AOF: Logs every write command. Safer — fsync options: every command,
   every second, or let the OS decide. Larger files, slower recovery.
   Can be rewritten/compacted periodically.

   Best practice: Use both. RDB for backups, AOF for durability.

Q: Caching strategies?
A: Cache-aside (most common): App checks cache first, on miss fetches
   from DB and stores in cache. I use this approach.

   Write-through: Write to cache AND DB simultaneously. Always consistent
   but slower writes.

   Write-behind: Write to cache only, async persist to DB later. Very
   fast writes but risk of data loss.

   Read-through: Cache itself fetches from DB on miss. Cleaner code but
   less control.

Q: Eviction policies?
A: What Redis does when memory is full:
   - noeviction: return errors on new writes
   - allkeys-lru: evict least recently used across all keys
   - volatile-lru: LRU only among keys with a TTL set
   - allkeys-lfu: evict least frequently used
   - volatile-ttl: evict keys closest to expiration
   - allkeys-random: random eviction

Q: How to implement distributed locks?
A: Simple: SET key value NX EX timeout — atomic set-if-not-exists with
   expiry. I use this in ApplySurge for exclusive user sessions.

   For stronger guarantees: Redlock algorithm — acquire lock on a
   majority of N independent Redis nodes. But it's controversial due
   to clock skew issues. For truly strict distributed locking, consider
   ZooKeeper or etcd.

Q: What is cache stampede? How to prevent?
A: When a popular cache key expires, hundreds of concurrent requests all
   miss the cache and hit the database simultaneously.

   Solutions:
   1. Mutex lock: only one request fetches, others wait for cache refill
   2. Probabilistic early expiration: refresh slightly before actual expiry
   3. Background refresh: a cron job refreshes popular keys
   4. Serve stale data while refreshing in the background

--------------------------------------------------------------------------------
4.3 MONGODB
--------------------------------------------------------------------------------

Q: What is the aggregation pipeline?
A: A sequence of stages that process documents:
   $match (filter), $group (aggregate), $project (reshape fields),
   $sort, $limit, $lookup (left outer join), $unwind (flatten arrays),
   $addFields, $facet (run multiple pipelines in parallel).

   Each stage transforms documents and passes them to the next. MongoDB
   optimizes the pipeline — for example, pushing $match before $sort.

Q: Embedding vs Referencing?
A: Embedding: Store related data as nested subdocuments. Best when data
   is always accessed together, need atomic updates, and the nested
   data doesn't grow unboundedly.

   Referencing: Store a reference (like a foreign key) and look up
   separately. Better when related data is accessed independently,
   shared across documents, or very large.

   In my system, extracted data is embedded within each document record
   because you always need it together. But template definitions are
   referenced by ID because they're shared across many documents.

Q: How does MongoDB handle flexible schemas?
A: Documents in the same collection don't need to have the same structure.
   This is powerful for my use case — each template extracts different
   fields, so the extracted data has different shapes per document type.
   You can optionally enforce schema validation at the collection level
   using JSON Schema if you want guardrails.

Q: What indexes does MongoDB support?
A: Single field, compound (multiple fields), multikey (arrays), text
   (full-text search), geospatial (2d, 2dsphere), hashed (for sharding),
   TTL (auto-delete after time), and wildcard indexes.

   Index selection matters for performance — use explain() to verify
   your queries are using indexes, not doing collection scans.

Q: Explain replica sets.
A: A group of MongoDB instances with the same data. One primary handles
   all writes, secondaries replicate and handle reads (if configured).
   Automatic failover: if primary goes down, remaining nodes elect a
   new primary (requires majority vote).

   Write concern controls how many replicas must acknowledge a write.
   Read preference controls where reads go — primary, secondary, nearest.

Q: WiredTiger storage engine?
A: Default since MongoDB 3.2. Key features:
   - Document-level concurrency control (not collection-level locking)
   - Compression: snappy (default), zlib, zstd
   - Journaling for crash recovery (similar to WAL)
   - Checkpointing every 60 seconds

--------------------------------------------------------------------------------
4.4 POSTGRESQL
--------------------------------------------------------------------------------

Q: What is MVCC?
A: Multi-Version Concurrency Control. Each transaction sees a snapshot
   of data at its start time. Writes create new row versions, old
   versions are kept until they're no longer needed. This means readers
   never block writers and vice versa. Old versions are cleaned up by
   the VACUUM process.

Q: What is WAL in PostgreSQL?
A: Write-Ahead Logging. All changes are written to the log before the
   actual data files. If the system crashes mid-write, it replays the
   WAL to recover. Also used for streaming replication — WAL segments
   are sent to replica servers.

Q: PostgreSQL index types?
A: - B-tree: Default. Equality + range. Most common.
   - Hash: Equality only. Rarely used since B-tree is almost as fast.
   - GIN: Generalized Inverted Index. For full-text search, JSONB, arrays.
   - GiST: Generalized Search Tree. Geometric data, range types, and
     pgvector similarity search.
   - BRIN: Block Range Index. Very efficient for large sequential data
     like time-series.

   For my EventZity project, pgvector uses a GiST or IVFFlat index for
   approximate nearest neighbor search on embedding columns.

Q: What is VACUUM?
A: Reclaims storage from dead rows left behind by MVCC updates/deletes.
   Regular VACUUM marks space reusable within the table (doesn't shrink
   the file). VACUUM FULL rewrites the entire table to reclaim disk space
   but locks the table during the operation.

   autovacuum runs automatically. Also critical for preventing transaction
   ID wraparound — a catastrophic failure if vacuum doesn't run.

Q: Connection pooling?
A: PostgreSQL forks a new process per connection — expensive (~10MB each).
   Connection poolers like PgBouncer sit between the app and database:
   - Transaction mode: connection returned after each transaction (best
     for most apps)
   - Session mode: dedicated connection per client session

   Dramatically reduces database resource usage for apps with many
   short-lived queries.

--------------------------------------------------------------------------------
4.5 DOCKER
--------------------------------------------------------------------------------

Q: Containers vs VMs?
A: Containers share the host OS kernel. Lightweight (MBs), fast startup
   (milliseconds), less isolation.

   VMs include a full guest OS on top of a hypervisor. Heavy (GBs),
   slow startup (minutes), strong isolation.

   Use containers for microservices and dev environments. VMs when you
   need different OS or strict security isolation.

Q: Images vs Containers?
A: Image: Read-only template built from a Dockerfile. Composed of layers.
   Shareable via registries (Docker Hub, ECR, GCR).

   Container: Running instance of an image with a writable layer on top.
   You can run multiple containers from the same image. Ephemeral by
   default — data is lost when the container stops unless you use volumes.

Q: What are Docker layers?
A: Each Dockerfile instruction creates a layer. Layers are cached — if
   the instruction and its context haven't changed, the cached layer is
   reused. This is why you put frequently changing instructions (like
   COPY . .) LAST and stable instructions (like installing dependencies)
   FIRST. Minimizing layer count (combining RUN commands) also helps.

Q: Multi-stage builds?
A: Multiple FROM statements in one Dockerfile. Build in one stage with
   all build tools and dev dependencies, then COPY only the necessary
   artifacts to a minimal final stage.

   Example: First stage uses the full Python image with all build tools.
   Final stage uses python-slim with only production dependencies and
   the built code. Dramatically reduces final image size.

Q: CMD vs ENTRYPOINT?
A: CMD: Default command, can be overridden when running the container.
   ENTRYPOINT: Fixed command, arguments from docker run are appended.

   Combining: ENTRYPOINT ["python"], CMD ["app.py"] — default runs
   "python app.py", but user can override with "docker run img other.py".

   Always use exec form (["cmd"]) over shell form ("cmd") to avoid
   an extra shell process wrapping your app.

Q: Docker Compose?
A: Define and run multi-container applications. You specify all services,
   networks, volumes, and environment variables in a docker-compose.yml.
   docker-compose up spins everything up, down tears it down. Great for
   local development with multiple services — API + database + cache +
   workers all running together.

Q: Docker networking?
A: - Bridge (default): Containers on the same bridge network can
     communicate by container name
   - Host: Container shares the host's network — no port mapping needed
   - None: No networking
   - Overlay: Multi-host networking for Docker Swarm/Kubernetes

   In Compose, services on the same network can reach each other by
   service name as hostname.

--------------------------------------------------------------------------------
5.1 PYTHON
--------------------------------------------------------------------------------

Q: What is the GIL?
A: Global Interpreter Lock in CPython. Only one thread can execute Python
   bytecode at a time. It's there to protect reference counting (Python's
   memory management).

   Impact: Multi-threading doesn't help for CPU-bound tasks.
   Solutions:
   - multiprocessing for CPU parallelism (separate processes, no GIL)
   - asyncio for I/O concurrency (my FastAPI services use this)
   - Threading still helps for I/O-bound tasks because the GIL is
     released during I/O operations

Q: How does async/await work in Python?
A: async def creates a coroutine. await suspends the coroutine and gives
   control back to the event loop. The event loop runs other coroutines
   while this one waits for I/O (network call, disk read).

   It's single-threaded concurrency — one thread handles many operations
   by not blocking. My FastAPI handlers are async, so one process can
   handle hundreds of concurrent requests efficiently.

Q: Explain decorators.
A: Functions that wrap other functions to add behavior. The @decorator
   syntax is sugar for func = decorator(func).

   Example: a timing decorator wraps a function, records start time,
   calls the function, prints elapsed time, returns the result.

   Use functools.wraps to preserve the original function's name and
   docstring. Common uses: logging, auth checks, rate limiting, Celery
   task registration.

Q: Generators vs Lists?
A: List stores everything in memory upfront (eager evaluation).
   Generator produces values one at a time using yield (lazy evaluation).

   Generators are memory-efficient for large datasets — you process one
   item at a time instead of loading everything. Use for: processing
   large files line by line, infinite sequences, data pipelines.

Q: Python memory management?
A: Three mechanisms:
   1. Reference counting: Each object tracks how many references point
      to it. When count hits zero, it's deallocated immediately.
   2. Cyclic garbage collector: Handles reference cycles (A points to B,
      B points to A). Generational — checks new objects frequently, old
      objects less often.
   3. Private heap: Memory allocator with pooling for small objects
      (<512 bytes) to avoid frequent system calls.

Q: List vs Tuple?
A: List: Mutable, slightly more memory, can append/modify. Use when
   the collection might change.

   Tuple: Immutable, hashable (can be a dictionary key), slightly faster,
   less memory. Use for fixed data — function return values, dict keys,
   constants.

Q: How do dicts work internally?
A: Hash table. Keys are hashed to determine position. Open addressing
   for collision resolution. Since Python 3.7+, insertion order is
   preserved. Average O(1) lookup. Resizes when roughly 2/3 full.

Q: is vs ==?
A: "is" checks identity — are these the exact same object in memory?
   "==" checks equality — do these have the same value?

   Gotcha: Python caches small integers (-5 to 256), so "256 is 256"
   is True, but "257 is 257" might be False. Always use == for value
   comparison. Only use "is" for None checks (x is None).

Q: Context managers?
A: Implements __enter__ and __exit__ methods. Guarantees cleanup even
   if an exception occurs. Classic example: file handling — the file
   is always closed, even if an error happens while reading.

   You can create custom ones with the @contextmanager decorator from
   contextlib — write a generator with a try/finally around the yield.

Q: *args and **kwargs?
A: *args collects positional arguments into a tuple.
   **kwargs collects keyword arguments into a dictionary.

   Used when you don't know how many arguments a function will receive.
   Common in decorators, class inheritance, and wrapper functions.

Q: Shallow vs Deep copy?
A: Shallow copy: New outer object, but nested objects are still references
   to the originals. Changes to nested objects affect both copies.

   Deep copy: Recursively copies everything. Completely independent clone.

   Matters for nested mutable structures. For flat structures, shallow
   copy is fine and cheaper.

5.2 JAVASCRIPT / TYPESCRIPT
--------------------------------------------------------------------------------

Q: Explain the Event Loop.
A: JavaScript is single-threaded. The event loop handles async operations:

   1. Call stack executes synchronous code
   2. When async operations complete, callbacks go to task queues
   3. Microtask queue (Promises, queueMicrotask) has priority
   4. Macrotask queue (setTimeout, I/O callbacks) runs after microtasks

   After each macrotask, the engine drains ALL microtasks before picking
   the next macrotask. This is why Promises resolve before setTimeout(0).

Q: What are Promises?
A: An object representing the eventual completion or failure of an async
   operation. States: pending -> fulfilled or rejected. Chainable with
   .then() and .catch().

   Key combinators:
   - Promise.all: resolves when ALL resolve, rejects on first rejection
   - Promise.allSettled: waits for all, returns status of each
   - Promise.race: resolves/rejects with the first settled promise
   - Promise.any: resolves with first fulfilled (ignores rejections)

Q: Explain closures.
A: A function that "remembers" variables from its enclosing scope, even
   after that scope has returned.

   Example: a counter function returns an inner function. The inner
   function accesses and modifies the outer function's "count" variable.
   Each call to counter() creates a new closure with its own count.

   Use cases: data privacy, factory functions, React hooks, memoization.

Q: How does 'this' work?
A: Depends on how the function is called:
   - In a method: 'this' is the object that owns the method
   - Standalone function: 'this' is window (browser) or undefined (strict)
   - Arrow function: 'this' is inherited from the enclosing scope
   - With call/apply/bind: 'this' is explicitly set
   - With 'new': 'this' is the new instance being created

   Arrow functions are great when you want 'this' from the surrounding
   context — callbacks, event handlers.

Q: var vs let vs const?
A: var: Function-scoped, hoisted and initialized as undefined. Can be
   redeclared. Legacy — avoid using.

   let: Block-scoped, in Temporal Dead Zone until declaration. Can't
   redeclare. Use when you need to reassign.

   const: Block-scoped, TDZ, must initialize, can't reassign the
   reference. But object properties ARE still mutable. Use by default.

Q: TypeScript Generics?
A: Parameterized types for reusable, type-safe code. Instead of using
   "any", you use a type parameter that the caller specifies.

   Constraints let you restrict what types are allowed. Utility types
   like Partial<T>, Pick<T, K>, Omit<T, K>, Record<K, V> are built-in
   generic types for common transformations.

Q: Debounce vs Throttle?
A: Debounce: Execute only after N milliseconds of NO calls. Resets timer
   on each call. Use for: search input (wait until user stops typing).

   Throttle: Execute at most once per N milliseconds. Guarantees regular
   execution. Use for: scroll handler, resize handler, rate limiting.

Q: Event delegation?
A: Attach a single event listener to a parent element instead of one per
   child. Events bubble up from the target to the parent. Check
   event.target to identify which child was clicked.

   Benefits: fewer listeners (less memory), automatically works for
   dynamically added child elements.

PART 6: AI / LLM ENGINEERING
================================================================================

Q: What is RAG (Retrieval-Augmented Generation)?
A: Combines retrieval from a knowledge base with LLM generation.
   1. Chunk documents into pieces
   2. Generate vector embeddings for each chunk
   3. Store in a vector database (pgvector, Pinecone, Weaviate)
   4. At query time: embed the user's query, find similar chunks
   5. Inject those chunks into the LLM prompt as context
   6. LLM generates an answer grounded in the retrieved information

   Reduces hallucination by giving the model real data to work with.
   I used this concept in EventZity for vendor search.

Q: What are embeddings?
A: Dense vector representations of text in continuous space. Semantically
   similar texts produce similar vectors. I use OpenAI's embedding model
   (1536 dimensions) for vendor search — "affordable photographer in
   Delhi" becomes a vector that's close to vectors of budget photographers
   in Delhi. Cosine similarity measures closeness between vectors.

Q: Prompt engineering techniques?
A: - Zero-shot: Just instructions, no examples
   - Few-shot: Include examples of input -> expected output
   - Chain-of-thought: "Think step by step" — improves complex reasoning
   - Role prompting: "You are an expert document parser..."
   - Structured output: "Return JSON with these exact fields..."
   - Temperature: 0 for deterministic/factual, higher for creative tasks

   My extraction prompts use role prompting + structured output formatting.

Q: Fine-tuning vs RAG?
A: Fine-tuning: Modify model weights on custom data. Expensive, needs
   curated datasets, changes the model's behavior. Hard to update.
   Use when: you need a specific output format or writing style.

   RAG: Keep the model as-is, provide external knowledge at query time.
   Cheaper, easy to update (just update the vector store). Better for:
   factual Q&A, knowledge that changes frequently.

   "For most use cases, start with RAG. Fine-tune only if RAG isn't
   enough."

Q: What are hallucinations? How do you handle them?
A: When the model generates confident but incorrect information.
   Mitigations:
   1. RAG — ground responses in real data
   2. Temperature = 0 for deterministic outputs
   3. Constrained generation (JSON schema, function calling)
   4. Validation layer that checks LLM outputs against known rules
   5. Human-in-the-loop for high-stakes decisions
   6. Ask the model to cite sources or say "I don't know"

Q: How do you handle token limits?
A: Models have maximum context windows (4K to 1M+ tokens). Strategies:
   1. Chunking: split large documents, process separately
   2. Summarization: compress context before sending
   3. Map-reduce: process chunks independently, combine results
   4. Sliding window: overlap between chunks to maintain context
   5. Use models with larger context when available

   My extraction system processes individual documents (each fits in
   context) rather than trying to batch everything.

Q: What is prompt injection?
A: Malicious input that tries to override the system prompt's instructions.
   Direct injection: user crafts a prompt to bypass instructions.
   Indirect injection: malicious content hidden in retrieved documents.

   Defenses: Input sanitization, separate system/user message roles,
   output validation, guardrails. In my system, extraction prompts are
   system-defined (not user-editable), which limits the attack surface.

Q: What are AI Agents?
A: LLMs that can take actions — not just generate text, but use tools
   (search, code execution, API calls), plan multi-step workflows, and
   iterate on results. The LLM decides which tools to use and in what
   order.

   In my LangGraph system, each agent has specific capabilities
   (PDF analysis, template creation, validation design) and can
   call tools to accomplish its goals.

Q: What is function calling / tool use?
A: The model decides when to call a predefined function based on the
   user's request. You define the function schema (name, parameters),
   the model returns a structured call, your code executes it, and the
   result goes back to the model for the next step.

   This is how agents work — the LLM doesn't execute code directly,
   it outputs structured function calls that your application executes.
